# 原力OS 3×3 二阶演化框架

> `原力OS = 治理OS × 工作OS`  
> `二阶演化 = 治理OS 的求解框架 + 工作OS 的分类框架 + 系统层级框架`

这份文档不重写 [YUANLI_OS.md](./YUANLI_OS.md)，也不替代 [YUANLI_OS_3X3_AUDIT.md](./YUANLI_OS_3X3_AUDIT.md)。

它只回答一个问题：

`3×3 往下怎么继续长？`

一句话回答：

> 3×3 的下一层，不是把九宫格机械再切一层，而是为治理OS长出更具体的求解器，为工作OS长出更具体的分类器，再给整个系统补上一套清晰的层级关系。

## 1. 二阶演化的总图

原力OS的一阶结构是：

- `治理OS`
  - 递归进化
  - 全局最优
  - 人类友好
- `工作OS`
  - 主题层
  - 策略层
  - 执行层

原力OS的二阶结构是：

- `治理OS 的二阶展开`
  - 每一条治理原则，继续长成一套求解框架
- `工作OS 的二阶展开`
  - 每一层工作结构，继续长成一套分类框架
- `系统层级框架`
  - 把“谁负责治理、谁负责编排、谁负责能力、谁负责执行”彻底讲清楚

这意味着：

- 一阶结构回答 `世界怎么被切开`
- 二阶结构回答 `每一块具体怎么求解`

## 2. 治理OS 的二阶展开

治理OS不是三句价值观，而是三套不同的求解器。

### 2.1 全局最优：四向求解

全局最优不是一句“你要想全局”，而是一套固定的搜索方向。

#### 上：外部最优

优先问：

- GitHub 上有没有更成熟的最佳实践
- 官方文档里有没有明确推荐路径
- 公共世界里有没有可复用攻略、样板、运行方式

含义：

- 在自己动手之前，先看天上有没有已经存在的好解

典型动作：

- 先看 GitHub
- 先读官方文档
- 先看业内已经跑通的范式

#### 下：本地真相

优先问：

- 当前 repo 真相是什么
- 当前 runtime、contracts、task ledger、审计报告已经证明了什么
- 当前真实约束、边界、失败点是什么

含义：

- 不拿抽象想象替代本地事实

典型动作：

- 先看当前仓库
- 先看当前运行证据
- 先看当前目标和边界

#### 左：历史沉淀

优先问：

- 历史上已经沉淀过哪些最佳经验
- 过往审计、复盘、失败样本能不能复用
- 之前有没有已经确认过的 canonical path

含义：

- 不重新造轮子，不忘记系统已经付出的学费

典型动作：

- 先看历史审计
- 先看已验证经验
- 先看已有规则和复盘

#### 右：横向复用

优先问：

- 相邻组件有没有现成做法
- 相邻 skill、相邻 Agent、相邻系统是怎么做的
- 别的链路上已经成熟的结构，能不能迁移进来

含义：

- 不把一个问题只困在本格子里解决，而要看同系统其他地方是否已经有答案

典型动作：

- 看龙虾别的组件
- 看智能体或 skill 干同类任务的方式
- 看现有审计与治理资产能否横向萃取

#### 全局最优的标准问句

面对任何任务，先问四句：

1. `上`：外部世界有没有更成熟的好解？
2. `下`：本地真相和当前边界是什么？
3. `左`：历史沉淀里有没有已经付过学费的答案？
4. `右`：相邻系统、相邻 skill、相邻 Agent 有没有现成可迁移结构？

### 2.2 递归进化：四步闭环

递归进化不是“做完复盘一下”，而是一个固定闭环。

#### 第一步：复盘事实

先回答：

- 这轮到底发生了什么
- 哪一步成功了
- 哪一步失败了
- 真完成和假完成分界在哪里

目标：

- 不让结论跑在事实前面

#### 第二步：抽象结构

再回答：

- 哪些只是这次的偶然
- 哪些已经显示出稳定结构
- 哪些可以被抽成标准动作、标准判断或标准链路

目标：

- 把事件变成结构，把经验变成方法

#### 第三步：写回系统

再问：

- 该写回规则吗
- 该写回 skill 吗
- 该写回 Agent 设计吗
- 该写回 runtime、contract、playbook 吗

目标：

- 让这轮经验进入系统，而不是停留在人脑里

#### 第四步：推进下一轮

最后问：

- 下一轮最小但真实的进化动作是什么
- 是补一个字段、补一条路由、补一个 skill、还是补一条验证链

目标：

- 让进化不是口号，而是下一步动作

#### 递归进化的标准问句

1. 这轮真实发生了什么？
2. 这里面哪些已经能抽象成稳定结构？
3. 这些结构应该写回系统的哪一层？
4. 下一轮最小但真实的进化动作是什么？

### 2.3 人类友好：四项克制检查

人类友好不是“态度温柔”，而是系统克制。

#### 少打扰

优先问：

- 这件事能否少让人显式介入
- 是否只在登录、授权、支付、不可逆决策时打断人

核心：

- 人不是流程引擎

#### 少重做

优先问：

- 这件事是否减少了重复劳动
- 是否减少了返工、补上下文、反复改口径

核心：

- 不把 AI 做过的劳动重新甩回给人

#### 少耗算

优先问：

- 是否减少了 token 浪费
- 是否减少了机器算力和无效搜索
- 是否先复用已有结构，再考虑重做

核心：

- 克制算力就是克制系统熵增

#### 少隐性复杂度

优先问：

- 是否降低了回滚成本
- 是否降低了审校成本
- 是否降低了“解释为什么这样设计”的沟通成本

核心：

- 真正的人类友好，不只是少点几次按钮，而是少背几层隐性复杂度

#### 人类友好的标准问句

1. 这一步是否减少了人的显式介入？
2. 这一步是否减少了人的重复劳动？
3. 这一步是否减少了算力和搜索浪费？
4. 这一步是否降低了回滚、审校、解释复杂度？

## 3. 工作OS 的二阶展开

工作OS的二阶展开，不是继续发散，而是继续收敛。

### 3.1 主题层：两层主题库

主题层的任务，不是把所有主题都列出来，而是先看系统里真实出现过什么，再往上收敛。

#### 第一层：当前仓库已出现的主题族

按当前 repo 资产，可以先收敛出六类主题族：

1. `治理与协作管家`
   - 例如：AI大管家、guan-jia-claw、治理 Base、线程治理
2. `内容生产与传播`
   - 例如：公众号写作、短视频增长、内容赛马、小红书采样
3. `工作流与自动化编排`
   - 例如：Coze、Dify、Feishu、n8n、workflow race
4. `知识库与研究审计`
   - 例如：知识库主路径审计、知识底座、攻略与说明书求解
5. `创业四周期推进`
   - 例如：原力觉醒、品类独创、模式升维、模式壁垒
6. `基础设施与接线可见性`
   - 例如：worker 联通、transport、可视化、接线、部署验证

#### 第二层：更高一层的母主题

这六类主题族可以继续收敛为五个母主题：

1. `认知与治理`
2. `内容与流量`
3. `工具与自动化`
4. `商业与增长`
5. `协作与基础设施`

#### 主题层的标准动作

主题层以后做两件事：

1. 先判断当前任务属于哪个 `主题族`
2. 再判断这个主题族属于哪个 `母主题`

这样做的意义是：

- 不让主题无限膨胀
- 让战略讨论有统一口径

### 3.2 策略层：五层智力模型

你前面说的数据、特征、观点、洞察、全局最优，这五层不应该漂在外面，而应正式安放为 `策略层的内部求解引擎`。

#### 第一层：数据

- 事实
- 样本
- 原始记录
- 可验证对象

这里回答：

- 我们到底知道了什么

#### 第二层：特征

- 模式
- 分类
- 相关性
- 结构信号

这里回答：

- 这些事实显出了什么结构

#### 第三层：观点

- 基于特征形成的判断
- 立场
- 解释框架

这里回答：

- 我怎么看这些结构

#### 第四层：洞察

- 比一般观点更深一层的机制理解
- 能解释表象背后的因果

这里回答：

- 为什么真正的问题在这里

#### 第五层：全局最优

- 把洞察继续推进成更大的取舍判断
- 不是只解释问题，而是决定下一步最优落子

这里回答：

- 当前已知条件下，什么才是最优一手

#### 策略层的标准动作

以后任何策略判断都要回答三件事：

1. 当前停在五层里的哪一层？
2. 目标要跃迁到哪一层？
3. 这个跃迁的证据来自哪里？

这样做的意义是：

- 不把“有观点”误当“有洞察”
- 不把“有洞察”误当“已经全局最优”

### 3.3 执行层：执行链族与通用骨架

执行层的二阶展开，要先把“所有事”分链，再把链收敛成通用骨架。

#### 第一层：标准执行链族

当前可以先归成五类：

1. `研究 / 审计链`
   - 例如：盘点、审计、说明书求解、知识库主路径判断
2. `内容生产链`
   - 例如：选题、写稿、改稿、标题、发布 package
3. `工作流构建链`
   - 例如：Dify / Coze / Feishu / n8n 方案、比较、部署
4. `接线 / 部署链`
   - 例如：账号配置、worker 注册、gateway 重载、验证
5. `治理闭环链`
   - 例如：route、review、close-task、sync、审计、规则写回

#### 第二层：通用执行骨架

所有执行链，最后都可以被压成同一条通用骨架：

1. `触发`
2. `澄清`
3. `路由`
4. `执行`
5. `验真`
6. `回贴`
7. `进化`

#### 执行层的标准动作

以后做任何事，先问两件事：

1. 这是五类执行链里的哪一类？
2. 目前卡在通用骨架的哪一段？

这样做的意义是：

- 不让执行链各说各话
- 让失败与卡点能被统一比较

## 4. 系统层级框架

3×3 的二阶演化，必须把系统层级讲清楚，否则“管家、Agent、skill、runtime”会一直混。

### 4.1 管家层

`管家层` 是顶层治理者。

它负责：

- 六判断
- 全局最优
- 边界
- 升级
- 闭环

它回答的是：

- 现在到底该做什么
- 该怎么调度下面的层
- 什么该升级，什么不该升级

当前代表：

- `AI大管家`
- `AI灵兽` 这种根层宪法角色

### 4.2 Agent 层

`Agent 层` 是中层任务家族编排器。

它不是能力原子，而是：

- 某类工作的一层稳定壳
- 某个主题族或任务家族的标准编排面

它负责：

- 承接一个任务家族
- 调用多个 skills
- 输出一条一致流程

关键定义：

> Agent 是中层壳，不是能力原子。

这层为什么要恢复出来：

- 因为只谈 skill，会导致系统“有能力但没壳”
- 因为很多高频工作，确实需要一个稳定的编排容器

但这层也要克制：

- 不回到旧式“大量 Agent catalogue”
- 不把每个想法都升格成 Agent

### 4.3 skill 层

`skill 层` 是可复用能力单元。

它继续沿用当前 repo 已经更成熟的分法：

- `root skills`
- `domain skills`
- `utility skills`

它负责：

- 某个能力点的稳定复用
- 某个领域动作的明确边界
- 被 Agent 或管家层按需调用

关键定义：

> skill 是能力原子，不是流程总控。

### 4.4 runtime / scripts / contracts 层

这是最底层执行基座。

它负责：

- 真实动作
- 真实状态
- 验真
- 留证

它包含：

- runtime
- scripts
- hooks
- contracts
- state
- artifacts

关键定义：

> runtime/scripts/contracts 负责把抽象判断变成真实可执行、可验证、可回滚的对象。

### 4.5 当前 repo 的兼容口径

当前 repo 的运营现实仍然是 `skill-first`。

所以要明确：

- 这次演化恢复 `Agent`，是恢复它的 `概念和编排层位置`
- 不是回到早期那种“列出大量 Agent 就等于系统完整”的表达

因此兼容口径固定为：

- `管家层`：负责治理
- `Agent 层`：负责任务家族编排
- `skill 层`：负责能力复用
- `runtime/scripts/contracts`：负责执行与验真

这四层之间不是替代关系，而是：

`管家定方向，Agent 组织流程，skill 提供能力，runtime 落真实动作。`

## 5. 两个固定示例

### 5.1 全局最优的四向求解示例

假设现在要做“飞书知识库主路径审计”。

先走四向求解：

- `上`
  去看 GitHub、官方文档、公开攻略，看有没有更成熟的知识库同步 canonical path
- `下`
  看当前 repo 已经有哪些脚本、哪些 README、哪些接线限制、哪些失败证据
- `左`
  看历史审计、历史 route、历史 close-task 里有没有已经确认的主路径
- `右`
  看相邻 skill、相邻治理链、相邻知识库任务是怎么做的

最后才落结论：

- 不是“我觉得哪个顺手”
- 而是“这四个方向都看过以后，当前最稳的 canonical path 是什么”

### 5.2 公众号内容系统的二阶落法

把“公众号内容系统”放进二阶框架：

#### 主题层

- 先归入 `内容生产与传播`
- 再归入母主题 `内容与流量`

#### 策略层

- `数据`
  阅读、反馈、样文、目标读者
- `特征`
  哪些题材更能形成稳定传播与认知沉淀
- `观点`
  当前应该做深度嵌入式分析，而不是追短热点
- `洞察`
  真正差异不在写作速度，而在主题判断和结构稳定性
- `全局最优`
  现阶段应该把公众号作为认知资产建设，而不是纯流量工具

#### 执行层

- 链族：`内容生产链`
- 骨架：
  - 触发：出现灵感或问题
  - 澄清：明确目标读者和目标结果
  - 路由：选择相关 Agent 和 skills
  - 执行：选题、大纲、初稿、改稿、标题、package
  - 验真：看证据、看质量、看目标 fit
  - 回贴：形成对人可用的发布或复盘对象
  - 进化：写回文风 DNA、标题经验、选题规则

## 6. 最后收敛

原力OS 3×3 的二阶演化，不是把框架变复杂，而是把框架变具体。

收敛成一句话：

> 一阶 3×3 解决的是“世界怎么切”，二阶 3×3 解决的是“每一块具体怎么求解、怎么分类、怎么落到系统层级”。

如果继续收敛成最短表达，就是：

- `治理OS` 长成求解器
- `工作OS` 长成分类器
- `系统层级` 长成执行秩序

三者合在一起，原力OS 才不会停留在理念层，而会进入真正可操作的系统层。
