# 原力OS 公众号实战应用说明书

> `原力OS = 治理OS × 工作OS`
>
> 这份文档回答一个更具体的问题：`原力OS 在公众号任务上怎么真正跑起来`

## 1. 这份文档解决什么问题

前面的总纲、二阶演化和审计，已经把 `原力OS` 讲清楚了。

但它们解决的更多是：

- 世界怎么切
- 每一块怎么求解
- 系统目前成熟到什么程度

这份文档解决的是第四个问题：

> 当你真的要写一篇公众号文章时，这套系统应该怎么落地。

因此，这不是理论补丁，而是一份 `实战应用说明书`。

这份说明书固定输出 5 个标准产物：

1. `Theme Brief`
2. `Strategy Card`
3. `Execution Run Sheet`
4. `Publish Pack`
5. `Evolution Note`

这 5 个对象合起来，公众号任务才算真正穿过了 `治理OS × 工作OS`。

### 1.1 六判断

- `自治判断`
  高自治。现有 `原力OS` 文档、公众号 skill 链和 pilot 资产已经足够支撑，不需要额外发明新结构。
- `全局最优判断`
  当前最优不是继续解释理念，而是把公众号任务压成一套可复制样板。
- `能力复用判断`
  优先复用现有 skill 链：`wechat-style-profiler`、`wechat-topic-outline-planner`、`wechat-draft-writer`、`wechat-title-generator`。
- `验真判断`
  每一层都必须有输入、输出、停止条件和验证点，不能只写“方法很好”。
- `进化判断`
  这份文档本身应该成为第一个高频任务家族的 playbook。
- `当前最大失真`
  现有文档解释了系统，但还没有把“一件具体事怎么跑成闭环”讲透。

## 2. 案例设定

为了把这套系统讲透，这份说明书固定使用一个代表性案例：

- `案例题目`
  为什么 AI 工具越多，团队反而越累，真正缺的是一套工作 OS
- `目标读者`
  创业者、管理者、职业人
- `文章目标`
  建立认知资产，带出方法论辨识度
- `主题族`
  内容生产与传播
- `母主题`
  内容与流量
- `文风样板`
  默认引用仓库已有的刘润公众号风格 DNA：
  `clawd-workspace/guan-jia-claw-workspace/runtime-artifacts/pilots/yuanli-openclaw-cold-thinking/01-style-dna.md`

### 2.1 两条路径

这份说明书明确区分两条路径。

#### Canonical Path

适合高质量、可复用、可积累资产的任务。

顺序固定为：

1. `wechat-style-profiler`
   只有在缺少 Style DNA 时触发
2. `wechat-topic-outline-planner`
3. `wechat-draft-writer`
4. `wechat-title-generator`

这条路径的特点是：

- 结构清晰
- 证据更强
- 更适合沉淀资产
- 更适合团队协作与长期复用

#### Fast Path

适合低严谨、单次快速出稿。

固定工具为：

- `wechat-article-writer`

这条路径的特点是：

- 更快
- 更省步骤
- 但治理深度弱
- 不作为这份说明书的主路径

### 2.2 人类只在这些节点介入

为了保持 `人类友好`，这份 playbook 只保留三种必要介入：

1. 主题层的最终进入判断
2. 大纲确认或重大角度取舍
3. 发文前的最终主观发布判断

除此之外，其他步骤都应由系统自行接住。

## 3. 治理OS 怎么落到公众号任务

治理OS 不是抽象价值观，而是公众号任务里的三套求解器。

### 3.1 全局最优：四向求解

在公众号任务里，`全局最优` 不能等于“我对这个题有感觉”。

它必须至少看四个方向。

#### 上：外部最优

先看公开世界里有没有更优的打法：

- GitHub 或公开 repo 里有没有成熟的内容工作流
- 官方文档或方法论里有没有更好的写作框架
- 公开优质作者最近怎么切这个题
- 有没有更成熟的标题、结构、证据组织方式

这一步解决的是：

> 不闭门造车。

#### 下：本地真相

再看当前任务的本地限制：

- 当前仓库里有哪些 skill 已经可用
- 当前有没有现成的 Style DNA
- 当前文章的目标是什么
- 当前读者是谁
- 当前有没有时间、事实、证据、平台约束

这一步解决的是：

> 不脱离现场。

#### 左：历史沉淀

再看历史上已经验证过什么：

- 过去的公众号 pilot 是怎么跑的
- 过去的审计和复盘发现了哪些失真
- 哪些标题风格、结构打法已经被验证有效
- 哪些失败样本说明什么不要再做

这一步解决的是：

> 不重复踩坑。

#### 右：横向复用

最后看相邻系统怎么做：

- 相邻 skill 是否已有可复用边界
- 相邻 Agent 是否已有稳定编排
- 相邻内容任务，比如短视频、赛马、审计报告，是否有可借用的验证与回贴方式

这一步解决的是：

> 不重复造轮子。

### 3.2 递归进化：四步闭环

在公众号任务里，递归进化不是“写完以后想一想”，而是强制写回系统。

#### 复盘事实

- 这轮选了什么主题
- 用了什么角度
- 哪些段落写得稳
- 哪些地方返工最多
- 哪些证据不足

#### 抽象结构

把一次性的经验，抽象成结构性经验：

- 什么选题更适合当前读者
- 什么结构更适合这个主题
- 什么标题风格更契合这个作者
- 什么验证点最容易漏

#### 写回系统

明确写回到哪里：

- 写回 `Style DNA`
- 写回 `选题规则`
- 写回 `标题规则`
- 写回 `执行清单`

#### 推进下一轮

定义下一轮最小真实进步：

- 少一次返工
- 少一次无效搜索
- 多一个稳定标题套路
- 多一个高复用 section 模板

### 3.3 人类友好：四项克制检查

公众号任务最容易出现一种假自动化：

> 看起来 AI 做了很多，实际上人只是在接更多中间稿。

所以必须过这四项检查。

#### 少打扰

- 不要在可以自动判断的节点反复问人
- 只有主题取舍、大纲确认、最终发布才找人

#### 少重做

- 不要没确认结构就写初稿
- 不要没确定 Style DNA 就硬模仿
- 不要标题和正文脱钩后再返工整篇

#### 少耗算

- 优先复用已有 DNA、已有证据、已有模板
- 不做大规模无边界搜索
- 不让多个 skill 重复完成同一子任务

#### 少隐性复杂度

- 让每一步产物都可回滚
- 让谁负责什么保持清楚
- 让验证点前置，而不是发文前一次性爆炸

## 4. 工作OS 三层怎么走

工作OS 负责把公众号任务拆成 `主题层 -> 策略层 -> 执行层`。

### 4.1 主题层

#### 主题层回答什么

- 这篇文章值不值得写
- 它是不是当前阶段应该进入主战场的内容
- 它服务的是品牌、销售、认知资产，还是纯热点流量

#### 主题层输出什么

输出固定为 `Theme Brief`，至少包含：

- 主主题
- 业务目标
- 为什么现在写
- 不写会损失什么
- 哪些角度不进入

#### 主题层停止条件

出现以下任一情况，任务应终止在主题层：

- 主题不服务当前阶段主战场
- 读者和业务目标说不清
- 这个题只能制造热点，不能沉淀资产
- 和更高优先级主题相比明显不优

#### 主题层的治理追问

- `递归进化`
  这个主题除了产出一篇文章，还会不会长出新的能力与复利？
- `全局最优`
  现在写这篇，真的比把资源投给其他主题更优吗？
- `人类友好`
  这个主题会让团队更聚焦，还是把人拖进高投入低复利的流水线？

### 4.2 策略层

#### 策略层回答什么

- 这篇文章准备怎么赢
- 目标读者真正会被什么打动
- 应该走什么结构与承诺
- 证据与风险边界在哪

#### 策略层输出什么

输出固定为 `Strategy Card`，至少包含：

- 当前停留层级
- 目标跃迁层级
- 证据来自哪里
- 推荐角度
- 结构打法
- 风险边界

#### 策略层的五层智力模型

1. `数据`
   有哪些事实、反馈、样文、读者信号、过往表现
2. `特征`
   从这些数据里能看出哪些稳定模式
3. `观点`
   这篇文章现在应该主张什么
4. `洞察`
   真正的矛盾在哪里
5. `全局最优`
   这一篇文章最值得占领的认知位置是什么

#### 策略层停止条件

出现以下任一情况，不能进入执行层：

- 观点和证据脱节
- 文章目标和读者不匹配
- 结构打法只是好看，不服务核心矛盾
- 风险边界没讲清楚

#### 策略层的治理追问

- `递归进化`
  这套打法会不会沉淀成以后还能复用的判断框架？
- `全局最优`
  这套策略是在解决核心矛盾，还是只优化局部表象？
- `人类友好`
  这套策略是在减少返工，还是在制造更多协调和审校成本？

### 4.3 执行层

#### 执行层回答什么

- 今天这篇文章怎么稳定做成
- 每一步由谁承接
- 每一步如何验证

#### 执行层输出什么

输出固定为 `Execution Run Sheet`。

#### 执行层通用骨架

1. `触发`
   捕捉问题、线索、灵感
2. `澄清`
   明确目标读者、文章目标、边界
3. `路由`
   选择 canonical path 还是 fast path，决定 skill 链
4. `执行`
   大纲、初稿、改稿、标题、发布包
5. `验真`
   检查事实、结构、文风、标题真实性
6. `回贴`
   形成对人可用的发文对象
7. `进化`
   写回规则与经验

#### 执行层停止条件

出现以下任一情况，必须停下修正：

- 没有确认大纲就直接起稿
- Style DNA 缺失却还在硬写
- 事实不足却没有标记 `[待补充证据]`
- 标题承诺超出正文真实内容

#### 执行层的治理追问

- `递归进化`
  这条链路每跑一轮，是否都变得更稳、更准、更省？
- `全局最优`
  这一步是真的推动结果，还是只制造进度幻觉？
- `人类友好`
  这一步是否真实减少了人的体力、时间和 token 消耗？

## 5. 管家 -> Agent -> skill -> runtime 在这件事里的分工

### 5.1 管家层

`管家层` 负责：

- 做六判断
- 决定这篇文章该不该进入
- 设定边界
- 选择 canonical path 还是 fast path
- 做最终闭环

在这件事里，管家不是写稿的人，而是 `治理者`。

### 5.2 Agent 层

这轮只恢复一个概念性中层：

- `公众号内容 Agent`

它不是新的 skill 清单，而是一层任务家族编排壳。

它负责：

- 接住一个公众号任务
- 调用合适的 skill
- 把产物串成统一流程

在这件事里，Agent 不是能力原子，而是 `流程组织者`。

### 5.3 skill 层

`skill 层` 负责能力复用。

在 canonical path 里，边界固定为：

- `wechat-style-profiler`
  负责文风资产，不负责选题与正文
- `wechat-topic-outline-planner`
  负责角度和大纲，不负责正文与标题定稿
- `wechat-draft-writer`
  负责初稿，不负责主题判断和标题策略
- `wechat-title-generator`
  负责标题，不负责大纲和正文

`wechat-article-writer` 只保留为快速单次模式，不作为主路径。

### 5.4 runtime / scripts / contracts 层

最底层只负责：

- 真实动作
- 真实状态
- 验真
- 留证

它不负责主题判断，也不负责策略取舍。

因此这四层的关系是：

> 管家定方向，Agent 组织流程，skill 提供能力，runtime 落真实动作。

## 6. 完整案例走读

下面用一篇完整案例，把这套系统真正走一遍。

### 6.1 Theme Brief

#### 主主题

解释为什么很多团队上了越来越多 AI 工具，却没有换来更高效率，真正缺的是一套工作 OS。

#### 业务目标

- 建立关于 `工作 OS` 的认知资产
- 在创业者和管理者读者心里建立方法论辨识度
- 为后续关于 `原力OS`、组织协作、AI 落地的内容主题打基础

#### 为什么现在写

- 当前市场讨论大量聚焦模型、工具和效率神话
- 但读者已经开始真实感受到“工具越多，组织越乱”的反作用
- 这是一个可以从表层热度切入、再落到更深判断的时间点

#### 为什么值得进入主战场

- 这个主题能连接 `内容与流量`、`认知与治理`、`工具与自动化`
- 它不是一次性热点，而是能沉淀后续内容母题
- 它适合作为 `原力OS` 的公众解释入口

#### 哪些角度不进入

- 不做纯工具测评
- 不做模型排行榜
- 不做空泛的 AI 焦虑抒情文
- 不把文章写成技术手册

#### 主题层停止判断

本题继续进入下一层，因为：

- 有明确读者
- 有明确业务目标
- 能沉淀长期母题
- 比单纯追工具热点更接近全局最优

### 6.2 Strategy Card

#### 当前停留层级

`观点层`

当前已有一个明确观点：

> 工具越多，不一定越高效；没有工作 OS，组织只会更忙。

#### 目标跃迁层级

`洞察层 -> 全局最优层`

要把文章从“观点正确”推进到“洞察清楚”，最终推进到“认知位置最优”。

#### 证据来自哪里

- `数据`
  当前团队普遍使用多个 AI 工具但协作摩擦没有下降
- `特征`
  真正的共性问题不是工具少，而是主题不清、策略不清、链路不清
- `观点`
  AI 时代缺的不是更多工具，而是工作 OS
- `洞察`
  很多团队以为自己在做 AI 落地，实际上只是在把旧混乱搬到新工具上
- `全局最优`
  这篇文章最该占的认知位置，不是“推荐几个工具”，而是“重新定义团队为什么会被工具反噬”

#### 推荐角度

推荐主角度：

> 不是 AI 没用，而是很多团队根本没有一套工作 OS，结果把工具当成了组织能力。

#### 读者承诺

读者看完后，至少要获得三件东西：

1. 能看懂为什么工具变多不等于效率变高
2. 能理解 `主题层 / 策略层 / 执行层` 的差异
3. 能把“工作 OS”当成一个可操作概念，而不是抽象口号

#### 结构打法

- 开头从常见误判切入
- 中段定义什么叫工作 OS
- 再拆 `主题层 / 策略层 / 执行层`
- 最后把问题收束到组织层面的判断升级

#### 证据需求

- 需要真实场景感，而不是只讲概念
- 需要给出一个团队常见工作链示例
- 需要解释为什么“局部高效”会导致“整体更乱”

#### 风险边界

- 不把文章写成反 AI 宣言
- 不把“工作 OS”写成玄学黑话
- 不承诺虚假的立即收益

### 6.3 主大纲与备选大纲

#### Primary Outline

##### Section 1

- `Section Objective`
  让读者先看到自己熟悉的问题
- `Core Claim`
  很多团队并不缺 AI 工具，缺的是把工具放进协作系统的能力
- `Evidence Need`
  工具越来越多、沟通越来越碎、返工越来越多的真实组织感
- `Transition`
  所以问题可能不在工具，而在系统

##### Section 2

- `Section Objective`
  定义什么叫工作 OS
- `Core Claim`
  工作 OS 不是工具栈，而是一套把事情分层、分工、分闭环的系统
- `Evidence Need`
  给出“同一任务里主题、策略、执行混在一起”的例子
- `Transition`
  既然它是系统，就必须先看系统分成哪几层

##### Section 3

- `Section Objective`
  解释三层结构
- `Core Claim`
  主题层决定打什么仗，策略层决定怎么赢，执行层决定怎么稳定做成
- `Evidence Need`
  用公众号或团队协作任务举例
- `Transition`
  但只有工作 OS 还不够，因为方向也会错

##### Section 4

- `Section Objective`
  引入治理OS
- `Core Claim`
  递归进化、全局最优、人类友好，决定了这套系统是不是越跑越强
- `Evidence Need`
  给出局部高效但整体更乱的反例
- `Transition`
  所以真正的 AI 落地，不是多一个工具，而是多一套操作系统

##### Section 5

- `Section Objective`
  收束与升维
- `Core Claim`
  AI 时代，团队真正的分水岭不是谁先装工具，而是谁先建立工作 OS
- `Evidence Need`
  给读者一个可带走的判断框架
- `Transition`
  结尾收束

#### Backup Outline

备选打法：

- 从一次典型团队返工事故切入
- 中段追溯返工不是执行问题，而是主题和策略前置缺失
- 最后再提出工作 OS

备选方案更有故事感，但理论解释的稳定性较弱，所以不作为主路径。

### 6.4 Execution Run Sheet

| 步骤 | 输入 | 动作 | 产物 | 验真点 | 责任层 |
| --- | --- | --- | --- | --- | --- |
| `触发` | 市场感受、内容灵感、近期观察 | 判断是否值得进入主题层 | 初始题目句 | 不是纯情绪，不是纯热点 | 管家 |
| `澄清` | 题目句、目标读者、文章目标 | 补齐约束与边界 | Brief Note | 读者、目标、边界说得清 | 管家 |
| `路由` | Brief Note、已有资产 | 选择 canonical path | 路由结论 | Style DNA 是否已存在 | 管家 + Agent |
| `执行一` | 题目、资料、读者目标 | 跑选题与大纲 | Theme Brief + Outline | 结构先于正文 | skill |
| `执行二` | 大纲、Style DNA、资料 | 跑初稿 | Draft v1 | 不编事实，缺证据标记 `[待补充证据]` | skill |
| `执行三` | Draft v1、目标读者、文风 | 跑标题 | Title Candidates x8 | 标题承诺与正文一致 | skill |
| `验真` | 全部产物 | 核查主题、结构、文风、标题真实性 | Verification Note | 是否真的服务文章目标 | 管家 |
| `回贴` | 已验证稿件 | 形成发文对象 | Publish Pack | 可以直接给人做最终发布判断 | Agent |
| `进化` | 本轮全链路 | 写回经验 | Evolution Note | 是否形成可复用规则 | 管家 |

#### 执行层降级规则

- `没有 Style DNA`
  先触发 `wechat-style-profiler`，再进入正文
- `主题不成立`
  停在 Theme Brief，不进入正文
- `资料不足`
  初稿中显式标记 `[待补充证据]`
- `标题过度承诺`
  回退到标题阶段，不允许强行发布

### 6.5 Draft v1

很多团队，最近都在做一件事。

装工具。

这个月装一个写作工具，下个月装一个会议助手，再下个月装一个自动化平台。看上去，什么都有了。可真正开始干活的时候，很多人却发现了一件奇怪的事。

工具越多，团队反而越累。

为什么？

因为大多数团队缺的，从来不是工具。

缺的是一套工作 OS。

什么叫工作 OS？

不是工具清单。不是流程图。更不是一句“我们以后都用 AI 干活”。

工作 OS，是一套把事情真正拆清楚的系统。它至少要回答三件事。

第一，这件事值不值得做。

第二，这件事准备怎么赢。

第三，这件事今天怎么稳定做成。

这三件事，对应三层。

主题层，策略层，执行层。

先说主题层。

很多团队一上来就问，怎么写，怎么做，怎么提效。可他们很少先问一句，这件事值不值得做。

比如，团队要做一篇内容。有人在想标题，有人在搜资料，有人在催初稿。看上去大家都很忙。但如果最前面没有人回答一个问题，这篇内容到底服务什么，服务品牌，服务销售，还是服务认知资产，那么后面所有努力，都可能只是高质量内耗。

这就是没有主题层。

再说策略层。

假设主题是对的。接下来还要回答，怎么赢。

是追热点，还是做深度分析？是打情绪，还是打判断？是让读者看完觉得热闹，还是让读者看完真的多拿走一个框架？

如果这些问题不提前想清楚，执行层越快，偏得越快。

最后才是执行层。

执行层当然重要。资料怎么整理，大纲怎么搭，初稿怎么写，标题怎么选，发布后看什么数据，这些都属于执行层。

但很多团队的问题恰恰在这里。

他们把所有问题，都误判成执行问题。

写得不够快，再上一个工具。

改得不够顺，再换一个工具。

协作有摩擦，再接一个工具。

结果是什么？

局部越来越高效，整体越来越混乱。

为什么会这样？

因为工具能放大动作，但不能替你决定方向。

它能把一个错误动作做得更快，却不能保证这个动作本身值得做。它能把一段文案改得更顺，却不能替你判断这篇文章到底该不该写。它能生成十个标题，却不能替你确认这十个标题是不是在服务真正的目标。

所以，AI 时代真正稀缺的，不是工具数。

而是治理能力。

也就是说，除了工作 OS，你还需要治理 OS。

治理 OS 至少有三个要求。

递归进化。每做完一篇，不只是发出去，而是留下文风规则、标题经验、选题判断。下一次要更稳，而不是重新来过。

全局最优。不要只看某一步是不是更快，要看这一步是不是在推动整体目标。否则，执行层会把错误的事做快，策略层会把错误的仗打漂亮。

人类友好。不要让 AI 名义上接手了工作，实际上却把更多审校、返工和管理成本甩回给人。

你会发现，当一个团队同时有了工作 OS 和治理 OS，很多事情就变了。

内容不再只是写出来，而是能沉淀。

流程不再只是跑一遍，而是会变强。

工具也不再是越多越好，而是越合适越好。

所以，真正的分水岭，从来不是谁先装了更多 AI。

而是谁先建立了一套工作 OS。

没有这套系统，再多工具，也只是更高级的忙碌。

有了这套系统，工具才会真正变成组织能力。

#### Title Strategy

这篇文章的标题策略，不追求“纯工具推荐感”，而要突出反差和误判纠正。

#### Best Title Recommendation

推荐标题：

`AI工具越多，团队为什么反而越累`

原因：

- 问题足够真实
- 情绪不过度
- 能最大化吸引创业者和管理者点击
- 正文可以稳稳接住

#### Safer Alternative

`不是AI没用，是你的团队还没有工作OS`

#### Stronger Alternative

`我见过很多团队装满AI，结果却更忙了`

#### Eliminated Titles And Why

- 过于抽象的标题应淘汰，因为读者不会第一时间理解 `工作OS`
- 过于承诺“立刻提效”的标题应淘汰，因为正文没有承诺即时翻倍收益

### 6.7 Publish Pack

#### 摘要

很多团队以为自己缺的是更多 AI 工具，结果工具装得越多，组织反而越乱。问题不在工具，而在缺少一套工作 OS。主题层决定打什么仗，策略层决定怎么赢，执行层决定怎么稳定做成。只有同时补上治理 OS，AI 才会从局部提效真正变成组织能力。

#### 配图建议

1. `封面图`
   一边是堆满工具图标的杂乱工作台，一边是结构清晰的流程白板
2. `中段示意图`
   用简图画出 `主题层 / 策略层 / 执行层`
3. `结尾图`
   用一句收束句配简洁结构图：工具不是系统，系统才能把工具变成能力

#### 发文前检查项

1. 主题是否清楚服务认知资产目标

3. 是否存在任何未经证实却写成事实的句子

5. `读者反馈关键词`
   看大家记住的是工具推荐，还是工作 OS 这个概念

### 6.8 Evolution Note

#### 这轮获得了什么

- 明确了一个高复用母题：
  `工具越多，不等于组织能力越强`
- 明确了一个稳定结构：
  `问题现象 -> 概念定义 -> 三层拆解 -> 治理升维 -> 结尾收束`
- 明确了一个标题方向：
  `反差 + 真实组织痛点`

#### 这轮浪费了什么

- 如果没有主题层前置，很容易滑向工具盘点文
- 如果没有风险边界，很容易把 `工作OS` 写成抽象黑话
- 如果不提前定义标题策略，标题阶段容易和正文割裂

#### 应该写回系统的规则

##### 主题规则

- 能进入公众号主战场的题，必须同时满足热点入口和长期母题价值

##### 标题规则

- 优先用真实反差提问，而不是空泛大词

##### 风格规则

- 先命名，再解释，再举例
- 用问题推进，不用堆信息推进

##### 流程规则

- 结构先于正文
- 正文先于标题定稿
- 缺证据就显式标记，不硬编

#### 下一轮最小进步

- 把 `Theme Brief`、`Strategy Card`、`Execution Run Sheet` 固化成模板
- 对标题和结尾收束各补一份标准 rubric

## 7. 一页式 SOP

如果下次要再做一篇公众号文章，直接按这张单页执行。

### Step 1

先做 `Theme Brief`。

只回答四件事：

- 这篇值不值得写
- 它服务什么业务目标
- 为什么现在写
- 哪些角度不进入

如果说不清，直接停。

### Step 2

再做 `Strategy Card`。

固定走五层：

- 数据
- 特征
- 观点
- 洞察
- 全局最优

如果只能停在观点层，不要急着起稿。

### Step 3

选路径。

- 高质量复用场景：走 `canonical path`
- 快速单次场景：走 `fast path`

默认优先 `canonical path`。

### Step 4

跑执行链：

- 触发
- 澄清
- 路由
- 执行
- 验真
- 回贴
- 进化

### Step 5

强制验真。

至少检查：

- 主题有没有跑偏
- 结构有没有服务核心矛盾
- 文风有没有贴住 DNA
- 标题有没有过度承诺

### Step 6

写 `Evolution Note`。

固定回答：

- 这轮沉淀了什么
- 这轮浪费了什么
- 下一轮最小进步是什么

## 8. 常见失真与纠偏

### 8.1 一上来就写

`失真`
把所有问题都当成正文问题。

`纠偏`
先做 `Theme Brief` 和 `Strategy Card`，结构先于正文。

### 8.2 把工具当系统

`失真`
以为多接几个工具，组织就会自动升级。

`纠偏`
先建立三层结构，再决定工具放在哪一层。

### 8.3 把观点当洞察

`失真`
有一句正确的话，就以为策略成立。

`纠偏`
强制回答：证据来自哪里，洞察是什么，全局最优是什么。

### 8.4 AI 做得更多，人却更累

`失真`
AI 产出很多中间稿，但人接了更多返工。

`纠偏`
只保留必要人类介入点，并前置验证。

### 8.5 发完就结束

`失真`
文章发完了，但系统没有变强。

`纠偏`
每一轮都写 `Evolution Note`，把经验写回主题、标题、风格和流程规则。

## 9. 最后收敛

用 `原力OS` 做公众号，不是让 AI 帮你多写一点。

而是让每一篇文章都同时回答三件事：

- 这件事值不值得做
- 这件事准备怎么赢
- 这件事今天怎么稳定做成

再用三条治理线不断追问：

- 它有没有让系统递归进化
- 它是不是更接近全局最优
- 它是否真实对人类友好

到这里，公众号任务才从“内容生产”，升级成“系统化内容生产”。
